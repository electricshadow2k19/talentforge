================================================================================
Version: 2.0
Audience: Cursor agent / engineering team
Purpose: End-to-end build, harden, and deploy TalentForge on cloud with browser
         access, JD-driven resume optimization, temporal skill validation, and
         professional document output.

Tagline: AI-Powered Recruiter Operating System for Staffing Companies

NOT: Public job board · Candidate marketplace · LinkedIn clone · Full ATS replacement

Primary goal: Reduce recruiter submission effort from 60+ minutes to under 5
minutes while producing factually accurate, professionally formatted resumes.


================================================================================
1. PRODUCT SUMMARY
================================================================================

TalentForge is an internal recruiter productivity platform for staffing firms
that manage bench consultants and candidate databases.

Recruiters and admins use a web portal (browser) to:
  1. Manage candidates and multiple resume versions per candidate
  2. Provide a Job Description (paste or file upload)
  3. Provide a Resume (Word/PDF upload or paste text)
  4. Run JD-driven evaluation and optimization
  5. Download professionally formatted optimized resume (DOCX/PDF)
  6. Generate submission package, interview questions, email drafts
  7. Track submissions and audit activity

The system MUST optimize resumes against JD requirements by:
  - Adding relevant skills, tools, and responsibility bullets where truthful
  - Rewording and reordering existing content for ATS alignment
  - NEVER fabricating employers, dates, projects, certifications, or experience

The system MUST perform TEMPORAL DUE DILIGENCE:
  - Before attributing a tool/skill/version to a project or role, verify when that
    technology/version was actually released
  - Only place that skill/version on employment periods that START AFTER release
  - Example: PostgreSQL 16 released ~2023 (GA). If JD asks for Postgres 16,
    do NOT add it to a project that ended in 2021. Only add to roles/projects
    whose date range overlaps post-release usage window.

7. CORE PIPELINE — JD-DRIVEN EVALUATION & RESUME UPDATE
================================================================================

Every optimization run follows this ordered pipeline:

  STEP 1 — INGEST
    • Parse JD from text or file (PDF/DOCX/TXT)
    • Parse resume from text or file (DOCX/PDF/TXT)
    • Extract raw text + structure (sections, roles, date ranges)

  STEP 2 — JD ANALYSIS (source of truth)
    Extract structured JSON:
      - job_title
      - required_skills, preferred_skills
      - tools, technologies, versions (e.g. "PostgreSQL 16", "K8s 1.29")
      - responsibilities
      - certifications, experience years
      - keywords
    Ignore for scoring: visa, citizenship, gender, age, race, religion

  STEP 3 — RESUME ANALYSIS
    Extract:
      - candidate_name, title, contact
      - employment_history[]: company, role, start_date, end_date, bullets
      - skills, tools, certifications, education
      - parsed date ranges per role (normalize to YYYY-MM)

  STEP 4 — ATS SCORING (before optimization)
    Weighted model (must sum to 100%):
      Skills Match         35%
      Tools & Technologies 20%
      Responsibilities     20%
      Experience Alignment 15%
      Keyword Coverage     10%

  STEP 5 — GAP ANALYSIS
    Compare resume vs JD:
      - matched_skills, missing_skills
      - missing_tools, missing_keywords
      - weak_areas, strength_areas
      - actionable_suggestions

  STEP 6 — TEMPORAL DUE DILIGENCE (NEW — CRITICAL)
    For each JD requirement that includes a versioned tool/skill:

    6a. Resolve release date from Technology Release Registry:
        - Curated JSON DB: { "PostgreSQL 16": "2023-09", "Kubernetes 1.29": "2023-12", ... }
        - Fallback: LLM + web knowledge with confidence flag
        - Store: tool, version, release_date, confidence

    6b. For each employment block in resume:
        - Parse start_date, end_date
        - Determine eligible window for technology attribution

    6c. Placement rules:
        - MAY add tool/version to role IF role end_date >= release_date
          (or role is "Present" / current)
        - MUST NOT add to roles that ended before release_date
        - If no eligible role exists: add to Skills section ONLY with wording
          like "exposure in lab/personal" ONLY if resume already supports it;
          otherwise flag as "cannot add without fabrication" in Temporal Review

    6d. Output temporal_audit[]:
        { requirement, release_date, placed_in_role, company, dates, allowed: true/false, reason }

  STEP 7 — RESUME OPTIMIZATION (JD-driven)
    Allowed:
      - Reword existing bullets for keyword alignment
      - Reorder sections and bullets by relevance to JD
      - Insert missing skills into Skills section IF candidate already has them elsewhere
      - ADD responsibility bullet points to eligible roles that demonstrate JD tools/skills
        using ONLY facts inferable from existing resume content
      - Improve professional summary tied to JD title

    Forbidden:
      - Fake employers, dates, degrees, certifications, projects
      - Adding tools to pre-release employment periods (blocked by Step 6)
      - Inventing years of experience with a technology

    Optimizer runs iteratively until ATS score plateaus or max 2 passes.

  STEP 8 — ATS RE-SCORE (after optimization)
    Score optimized resume; expose before/after delta.

  STEP 9 — GENERATE ARTIFACTS
    - Submission package (summary, strengths, risks, skills matrix)
    - Interview questions (technical, scenario, behavioral, JD-specific)
    - Email drafts (vendor cover-letter style; no ATS score in vendor email)

  STEP 10 — PROFESSIONAL DOCUMENT BUILD
    - Build DOCX via structured section model (not raw text dump)
    - Build PDF from same structure
  - Validate: no duplicate headings, no overlap, spell-check optional

  STEP 11 — PERSIST
    - Save JobDescription, Submission, package_json, temporal_audit
    - Audit log entry

8. TECHNOLOGY RELEASE REGISTRY (TEMPORAL ENGINE)
================================================================================

8.1 PURPOSE
  Prevent anachronistic claims (e.g. Postgres 16 experience in 2019 role).

8.2 DATA SOURCE (layered)
  Layer 1 — Static registry file (backend/data/tech_releases.json):
    {
      "postgresql": {
        "16": { "release_date": "2023-09-14", "notes": "PG 16 GA" },
        "15": { "release_date": "2022-10-13" }
      },
      "kubernetes": { "1.29": { "release_date": "2023-12-13" } },
      ...
    }

  Layer 2 — LLM lookup with structured output (when not in registry):
    { "tool", "version", "release_date", "confidence": "high|medium|low" }

  Layer 3 — Manual override table in DB (admin can correct dates)

8.3 MATCHING LOGIC
  - Normalize JD tokens: "Postgres 16", "PostgreSQL 16", "PG16" -> same entity
  - Match employment date ranges with month granularity
  - "Present" / "Current" = today

8.4 UI — TEMPORAL REVIEW TAB
  Show table:
    Requirement | Release Date | Target Role | Role Dates | Allowed | Action Taken
  Highlight blocked placements in amber/red with explanation.


================================================================================
9. RESUME FORMATTING ENGINE
================================================================================

9.1 PROBLEM (current gaps to fix)
  - Overlapping blocks in preview
  - Redundant headings
  - Non-professional layout

9.2 SOLUTION — STRUCTURED RESUME MODEL
  Intermediate representation (not flat text):

    ResumeDocument:
      header: { name, title, contact_lines[] }
      summary: string
      skills: { category: string, items: string[] }[]
      experience: {
        company, role, location, start, end,
        bullets: string[]
      }[]
      education: { degree, school, year }[]
      certifications: string[]

  9.3 DOCX BUILDER RULES
    - Centered name (18pt bold), title (12pt), contact (10pt)
    - Section headings: ALL CAPS, bold, horizontal rule optional, ONCE only
    - Job header: Company | Role | Dates on one line; location on next if present
    - Bullets: hanging indent, consistent "•" or dash
    - Skills: comma-separated or 2-column layout; no duplicate skill tokens
    - Max 2 pages recommended; warn if exceeded

  9.4 PREVIEW
    - Frontend: docx-preview on generated blob OR server-side render
    - Must match downloadable DOCX

  9.5 VALIDATION LINTER (post-build)
    - assert no duplicate section names
    - assert every experience block has dates
    - assert no skill appears in experience before release_date (cross-check Step 6)
